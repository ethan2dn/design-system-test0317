# PD 마크업 프로세스 — FE 협업 체크리스트

## 개요

Product Designer(PD)가 바이브코딩으로 마크업까지 담당하는 프로세스를 도입하기 위해,
FE 개발자와 함께 검토하고 합의해야 할 항목들을 정리합니다.

---

## 1단계: FE에게 현재 PoC 리뷰 요청

### 1-1. 코드 품질 리뷰

PD가 만든 코드가 실제 프로덕션 코드베이스에 통합 가능한 수준인지 확인.

- [ ] **컴포넌트 구조**: 폴더 구성, 파일 분리 방식이 팀 컨벤션과 맞는가?
  - 현재: `src/components/{Name}/{Name}.jsx + index.js`
- [ ] **코드 스타일**: 네이밍, 포맷팅, 린팅 규칙 확인
  - ESLint / Prettier 설정이 필요한가?
- [ ] **CSS 방식**: 인라인 스타일 vs CSS Modules vs styled-components 등
  - 현재: 인라인 스타일 + CSS 변수(토큰) → FE 팀 기술 스택과 맞는지?
- [ ] **React 패턴**: state 관리, hook 사용, prop 설계가 적절한가?
- [ ] **접근성(a11y)**: 시맨틱 HTML, ARIA 속성 등 기본 수준 충족하는가?

### 1-2. 디자인 토큰 리뷰

- [ ] **토큰 네이밍**: `--color-{theme}/{category}/{name}` 구조가 기존 시스템과 호환되는가?
- [ ] **토큰 생성 방식**: `figma-tokens.json → generate-tokens-css.cjs → tokens.css` 파이프라인이 적절한가?
- [ ] **토큰 범위**: 컬러 254개, 타이포 51개 — 누락되거나 불필요한 토큰이 있는가?
- [ ] **다크모드**: `light/*` / `dark/*` 토큰 분리 방식이 기존 또는 계획된 방식과 맞는가?

### 1-3. 컴포넌트 리뷰

- [ ] **Button**: prop API (styleType, color, size, strength 등)가 적절한가?
- [ ] **NavigationBar**: type/device/layout 분기 로직이 맞는가?
- [ ] **Icon**: SVG inline 방식 vs SVG sprite vs 다른 방식?
- [ ] **Layout**: breakpoint, 패딩, max-width 규칙이 서비스와 일치하는가?

### 1-4. 에셋 관리 리뷰

- [ ] **아이콘 SVG**: 199개 아이콘이 Icon.jsx에 inline으로 들어가 있음 → 번들 사이즈 이슈?
- [ ] **Lottie JSON**: 에셋 파일 관리 방식 (CDN? 번들 포함?)
- [ ] **이미지**: 프로필/썸네일 기본 이미지 관리 위치

## 2단계: 프로세스 합의

### 2-1. PD ↔ FE 역할 경계

| 항목 | PD 담당 | FE 담당 |
|------|---------|---------|
| 디자인 토큰 정의 (피그마) | O | |
| 토큰 → CSS 변수 변환 | O | 리뷰 |
| UI 컴포넌트 마크업 | O | 리뷰 |
| 컴포넌트 인터랙션/로직 | 기본 상태 | 복잡한 로직 |
| API 연동 | | O |
| 상태 관리 (전역) | | O |
| 라우팅 | | O |
| 다국어 매핑 (텍스트 교체) | O | 리뷰 |
| 스토리북 Playground/Spec | O | |
| 접근성 심화 | | O |
| 성능 최적화 | | O |
| 코드 리뷰 & 머지 | | O |

### 2-2. 브랜치 / PR 전략

- [ ] PD가 작업할 브랜치 규칙 (e.g. `markup/page-name`)
- [ ] PR 리뷰 프로세스 — PD가 PR 생성 → FE가 리뷰 → 머지
- [ ] 자동 린팅/포맷팅 CI 설정 (PD 코드 품질 자동 보정)

### 2-3. 기술 스택 정렬

- [ ] 현재 PoC: React + Vite + Storybook + CSS 변수
- [ ] 실제 프로덕션은 어떤 스택인가? (Next.js? TypeScript? CSS-in-JS?)
- [ ] PD 마크업을 프로덕션에 통합하려면 어떤 변환이 필요한가?
- [ ] TypeScript 전환이 필요한가? (현재 JSX)

## 3단계: 프로세스 정립

### 3-1. PD 마크업 워크플로우

```
1. 피그마 디자인 완료
2. 피그마 MCP로 디자인 데이터 추출
3. 디자인 토큰 확인/업데이트
4. 바이브코딩으로 페이지 마크업
   - 기존 컴포넌트 활용
   - 토큰만 사용 (하드코딩 금지)
   - 반응형 레이아웃 적용
   - 다국어 코드 매핑
5. 디버그 패널 구성 (상태별 확인)
6. 스토리북 Playground + Spec 작성
7. PR 생성 → FE 리뷰
8. FE가 API 연동/로직 추가 → 머지
```

### 3-2. 품질 게이트

PD 마크업이 PR로 올라오기 전에 확인해야 할 최소 기준:

- [ ] 모든 텍스트가 다국어 코드로 매핑되어 있는가?
- [ ] 모든 컬러/타이포가 토큰을 사용하는가?
- [ ] 반응형이 동작하는가? (모바일/PC)
- [ ] 다크/라이트 모드가 동작하는가?
- [ ] 스토리북에서 모든 상태를 확인할 수 있는가?
- [ ] 피그마 디자인과 마크업이 일치하는가?

## 4단계: 확장 계획

### 4-1. 추가 컴포넌트 필요 여부

현재 만들어진 컴포넌트: Button, NavigationBar, Icon, Layout

추가로 필요할 수 있는 공용 컴포넌트:
- [ ] Input / TextField
- [ ] Modal / Dialog
- [ ] Toast / Snackbar
- [ ] Card (공용)
- [ ] Tab / SegmentedControl
- [ ] Checkbox / Radio / Toggle
- [ ] Select / Dropdown
- [ ] Divider
- [ ] Badge / Tag
- [ ] Bottom Sheet

→ 어떤 컴포넌트를 PD가 만들고, 어떤 것을 FE가 만들지 합의 필요.

### 4-2. 다국어 프로세스

- [ ] 현재: CSV 수동 다운로드 → i18n/index.js에 수동 추가
- [ ] 개선안: 구글 시트 API 자동 연동? 빌드 시 자동 생성?
- [ ] 서비스 계정 키 세팅 필요?

### 4-3. 디자인 토큰 자동화

- [ ] 피그마 토큰 변경 시 자동 반영 파이프라인
- [ ] Figma Variables API 또는 Tokens Studio 연동 검토

## FE에게 전달할 질문 리스트

1. 현재 PoC 코드를 봤을 때, 프로덕션 코드베이스에 통합 가능한가?
2. 통합하려면 어떤 변환/수정이 필요한가?
3. PD가 만든 마크업의 코드 품질 기준을 어느 수준으로 잡으면 되는가?
4. 인라인 스타일 방식을 유지해도 되는가, CSS 방식을 바꿔야 하는가?
5. 컴포넌트 prop API 설계를 FE가 먼저 정의하고 PD가 따르는 게 나은가?
6. PR 리뷰 시 FE가 집중적으로 볼 포인트는?
7. PD 마크업의 자동 품질 검사(lint, format)를 CI에 넣을 수 있는가?
8. 다국어/토큰 자동화에 FE 리소스를 투입할 수 있는가?
